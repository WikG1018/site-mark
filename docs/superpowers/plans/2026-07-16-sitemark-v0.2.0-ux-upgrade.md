# SiteMark v0.2.0 UX Upgrade Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver faster consecutive system-camera capture, persistent background photo processing, image previews, date filtering, global settings, and a 20% larger watermark without losing existing v0.1.0 user data.

**Architecture:** Flutter keeps UI, Drift persistence, routing, and workflow orchestration. A WorkManager-backed serialized queue runs an idempotent Dart `CaptureProcessor` in a headless isolate, while a repository-local `FlutterPlugin + ActivityAware` Pigeon bridge exposes MediaStore operations without requiring `MainActivity`. Rust remains the sole full-resolution image renderer.

**Tech Stack:** Flutter/Dart, Riverpod, GoRouter, Drift/SQLite, `workmanager 0.9.0+3`, `package_info_plus 10.2.0`, Pigeon, Kotlin, flutter_rust_bridge, Rust/image/imageproc, Android API 31-36.

class CaptureFilter {
  const CaptureFilter({this.projectId, this.year, this.month, this.day})
      : assert(month == null || year != null),
        assert(day == null || month != null);

  final String? projectId;
  final int? year;
  final int? month;
  final int? day;

  CaptureDateRange? get localRange {
    if (year == null) return null;
    final start = DateTime(year!, month ?? 1, day ?? 1);
    final end = day != null
        ? DateTime(year!, month!, day! + 1)
        : month != null
            ? DateTime(year!, month! + 1)
            : DateTime(year! + 1);
    return (start: start, end: end);
  }

  CaptureFilter selectProject(String? value) => CaptureFilter(
    projectId: value,
    year: year,
    month: month,
    day: day,
  );

  CaptureFilter selectYear(int? value) => CaptureFilter(
    projectId: projectId,
    year: value,
  );

  CaptureFilter selectMonth(int? value) => CaptureFilter(
    projectId: projectId,
    year: year,
    month: value,
  );

  CaptureFilter selectDay(int? value) => CaptureFilter(
    projectId: projectId,
    year: year,
    month: month,
    day: value,
  );
}

class SiteMarkSystemPlugin : FlutterPlugin, ActivityAware,
    PluginRegistry.ActivityResultListener,
    PluginRegistry.RequestPermissionsResultListener {
    private var api: AndroidSystemApi? = null
    private var messenger: BinaryMessenger? = null
    private var binding: ActivityPluginBinding? = null

    override fun onAttachedToEngine(binding: FlutterPlugin.FlutterPluginBinding) {
        messenger = binding.binaryMessenger
        api = AndroidSystemApi(binding.applicationContext)
        SiteMarkSystemApi.setUp(binding.binaryMessenger, api)
    }

    override fun onAttachedToActivity(binding: ActivityPluginBinding) {
        this.binding = binding
        api?.attachActivity(binding.activity)
        binding.addActivityResultListener(this)
        binding.addRequestPermissionsResultListener(this)
    }

    override fun onDetachedFromActivityForConfigChanges() = detachActivity()
    override fun onReattachedToActivityForConfigChanges(binding: ActivityPluginBinding) =
        onAttachedToActivity(binding)
    override fun onDetachedFromActivity() = detachActivity()
}

- [ ] **Step 3: Split capture coordination from processing**

Define processor results and error classification:

```dart
enum CaptureProcessResult { succeeded, alreadyComplete, retry, failed, missing }

final class CaptureProcessor {
  CaptureProcessor({
    required AppDatabase database,
    required PlatformServices platform,
    required ImagePipeline images,
    required CaptureOutputPaths outputPaths,
  });

  Future<CaptureProcessResult> process(String captureId);
}
```

Processing order is fixed:

1. Return `missing` if the record or project no longer exists.
2. Return `alreadyComplete` for `ready`.
3. Reject `pendingCamera` because foreground recovery owns it.
4. Increment `processingAttempts` in one database transaction.
5. Verify that original path, captured time, and photo number exist; permanent failure otherwise.
6. Calculate SHA-256 only when missing; when present, verify the current original against it.
7. Mark `rendering`, render to `rendered/<captureId>.jpg`, then publish using `photoNumber`.
8. Mark `ready` with the returned URI.
9. Return `retry` for IO/system failures when attempts are below 3; on attempt 3 mark `failed` and return `failed`.

Change capture outcome:

```dart
enum CaptureWorkflowOutcome { queued, cancelled, failed }
```

Make repeated worker transitions idempotent and explicitly permit regeneration and retry:

```dart
bool canTransitionTo(CaptureStatus next) =>
    next == this || allowedNext.contains(next);

// Keep the current pending/captured/rendering transitions, then use:
CaptureStatus.failed => {CaptureStatus.captured},
CaptureStatus.ready => {CaptureStatus.captured},
```

After `CameraOutcome.captured`, `CaptureWorkflow.capture` must call `markCaptured`, finish the camera target with `keepOriginal = true`, enqueue the ID, and return immediately. `recoverPendingCapture` performs the same captured-and-enqueue transition. `regenerateCapture` updates editable fields, resets attempts/state, enqueues, and returns the queued record without rendering inline.

Define an injectable adapter and production scheduler:

```dart
abstract interface class CaptureBackgroundScheduler {
  Future<void> initialize();
  Future<void> enqueue(String captureId);
  Future<void> reconcilePending();
  Future<void> retry(String captureId);
}

const captureProcessingTask = 'sitemark.processCapture';
const captureProcessingQueue = 'sitemark-render-queue';

@pragma('vm:entry-point')
void captureCallbackDispatcher() {
  Workmanager().executeTask((taskName, inputData) async {
    WidgetsFlutterBinding.ensureInitialized();
    if (taskName != captureProcessingTask) return true;
    final captureId = inputData?['captureId'] as String?;
    if (captureId == null || captureId.isEmpty) return true;
    await (_backgroundRustInitialization ??= RustLib.init());
    final database = AppDatabase();
    try {
      final result = await buildHeadlessCaptureProcessor(database).process(captureId);
      return result != CaptureProcessResult.retry;
    } finally {
      await database.close();
    }
  });
}
```

Declare `Future<void>? _backgroundRustInitialization;` at library scope so a reused headless engine initializes FRB exactly once.

Use a unique serial chain named `sitemark-render-queue`, append-or-replace semantics, input `{'captureId': captureId}`, and tag `capture:$captureId`. Return `false` only for transient retry; final failures return `true` so later work in the chain is not cancelled.

Initialize WorkManager before `runApp`, then run camera recovery followed by scheduler reconciliation from startup. Disable both in widget tests through provider overrides.
