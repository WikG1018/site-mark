# SiteMark v0.2.0 使用体验升级设计

## 背景与目标

SiteMark v0.1.0-alpha 已完成项目管理、系统相机拍摄、定位、Rust 全分辨率水印、MediaStore 发布和工程记录导出。首轮真机使用暴露出五个高频问题：水印文字偏小、拍摄表单不记忆上一条记录、照片渲染会阻塞下一次拍摄、记录没有图片预览、首页缺少全局设置和跨记录筛选。

v0.2.0 的目标是在不改变“无广告、无云端、系统相机、本地保存”产品边界的前提下，提高工程现场的连续拍摄效率和记录检索效率。成功标准如下：

- 水印排版整体放大 20%，文字可读且不裁切。
- 同一项目再次拍摄时，自动带入上一张的工作地点、工作内容和拍摄人，备注为空。
- 系统相机返回后不等待哈希、水印渲染和相册发布，用户可立即再次拍摄。
- 退出 App、进程回收或设备重启后，未完成照片仍能继续处理。
- 项目记录和全部记录均能看到缩略图，并按年、月、日筛选。
- 首页提供全局设置和全部记录入口。

## 已确认的产品决策

### 连拍

- 相机返回并确认原图有效后，前台只完成照片编号分配、记录状态持久化和后台任务入队。
- 成功入队后停留在拍摄表单，不自动再次拉起系统相机。
- 工作地点、工作内容和拍摄人保持当前值；备注立即清空。
- 相机取消时保留全部表单内容，不创建可见记录。
- 前台提示“已加入后台处理”，拍摄按钮恢复可用。

### 后台处理

- Android 使用 WorkManager 一次性持久任务；所有全分辨率照片进入同一串行队列。
- 一条任务执行：读取记录与项目、校验原图、计算或复用 SHA-256、渲染水印、更新 MediaStore 同名照片、标记完成。
- 任务必须可重复执行：记录已完成时直接成功；中断在 `captured` 或 `rendering` 时从数据库状态继续。
- 临时错误采用指数退避，最多自动尝试 3 次；超过次数后标记失败并提供人工重试。
- App 启动时同时恢复系统相机遗留记录，并重新调度遗漏的 `captured`/`rendering` 记录。
- 描述信息编辑后的水印重新生成也走同一后台队列。

### 预览与筛选

- 列表缩略图尺寸为 96×96，使用圆角裁切。
- `ready` 显示最终水印图；`captured`/`rendering` 显示原图和状态角标；失败时优先显示原图和错误角标；文件不存在时显示占位图。
- 详情页顶部显示大图，点击进入支持 1–4 倍缩放和拖动的全屏预览。
- 项目详情提供“年 → 月 → 日”级联筛选。
- 首页新增“全部记录”页面，提供“项目 → 年 → 月 → 日”级联筛选。
- 筛选日期采用手机本地时区的 `capturedAt`；旧记录缺少该值时回退到 `createdAt`。
- 父级条件变化时清空不再有效的子级条件；选项只展示实际存在的项目和日期。

### 全局设置

- 首页 AppBar 固定显示“全部记录”和“设置”图标；原“无广告”Chip 移至关于信息，避免小屏拥挤。
- 外观：跟随系统、浅色、深色。
- 语言：跟随系统、简体中文、English。
- 新项目默认水印：左下/右下、透明度 0.20–0.95、绿色/蓝色/橙色强调色。
- 全局水印默认值只在创建新项目时复制到项目记录，不追溯修改已有项目。
- 关于：应用名称、版本号、构建号、无广告、无账号、无云端、系统相机、本地保存、GitHub 仓库、隐私政策、Apache-2.0 和第三方许可。

## 技术设计

### 数据模型

Drift schema 从 2 升级到 3：

- 新建单例 `app_settings` 表，固定主键 `global`，保存 `themeMode`、`localeCode`、`defaultWatermarkPosition`、`defaultWatermarkOpacity`、`defaultWatermarkAccentColorArgb` 和 `updatedAt`。
- `captures` 新增 `processingAttempts INTEGER NOT NULL DEFAULT 0`，用于控制自动重试次数。
- 升级事务必须保留所有现有项目、照片编号、原图路径、哈希和已发布 URI，并插入默认设置行。
- 默认主题和语言均为 `system`；默认水印沿用 v0.1.0 的 `bottomLeft`、`0.78`、`0xff37c58b`。

新增查询模型：

```dart
class CaptureFilter {
  const CaptureFilter({this.projectId, this.year, this.month, this.day});

  final String? projectId;
  final int? year;
  final int? month;
  final int? day;
}

class CaptureSummary {
  const CaptureSummary({required this.capture, required this.projectName});

  final CaptureRecord capture;
  final String projectName;
}
```

记录列表不展示 `pendingCamera`，其余状态按 `capturedAt ?? createdAt` 倒序。数据库查询接收本地日期边界并转换为 `[startInclusive, endExclusive)`，避免字符串日期比较和月末错误。

### 后台任务边界

新增两个明确职责的 Dart 服务：

```dart
abstract interface class CaptureBackgroundScheduler {
  Future<void> initialize();
  Future<void> enqueue(String captureId);
  Future<void> reconcilePending();
  Future<void> retry(String captureId);
}

class CaptureProcessor {
  Future<CaptureProcessResult> process(String captureId);
}
```

`CaptureWorkflow` 只负责需要 Activity 的定位和系统相机阶段。相机成功后返回 `CaptureWorkflowOutcome.queued`。`CaptureProcessor` 只处理数据库中已经存在的照片，不启动相机、不请求定位，也不依赖 Widget 或 Riverpod 容器。

WorkManager 回调是顶层 `@pragma('vm:entry-point')` 函数，在独立 isolate 中依次初始化 Flutter binding、RustLib、Drift、输出路径和 MediaStore 桥接。任务以 `captureId` 为唯一输入，使用 `capture:<id>` 标签，全部追加到 `sitemark-render-queue` 唯一工作链。

### Android 桥接

当前 Pigeon API 由 `MainActivity.configureFlutterEngine` 手工注册，后台 FlutterEngine 无法访问。将它移动到仓库内路径依赖插件 `packages/sitemark_system_api`：

- `SiteMarkSystemPlugin` 实现 `FlutterPlugin` 和 `ActivityAware`。
- 插件附着 FlutterEngine 时立即注册 Pigeon；后台仅凭 application `Context` 即可执行 MediaStore 发布和删除。
- 附着 Activity 后注册相机结果和定位权限监听；脱离 Activity 时取消监听并使相机/定位调用明确失败。
- `MainActivity` 恢复为无自定义逻辑的 `FlutterActivity`。
- ContentProvider 与 `${applicationId}.capture` authority 由插件 Manifest 合并进应用。

### 状态和错误

沿用 `pendingCamera → captured → rendering → ready/failed` 状态机：

- `captured` 在界面上显示“等待处理”。
- `rendering` 显示“处理中”。
- 临时失败且尝试次数小于 3 时保持可恢复状态并由 WorkManager 重试。
- 最终失败转为 `failed`，保存面向用户的简短原因。
- 人工重试将尝试次数清零、状态恢复为 `captured` 并重新入队。
- 原图不存在、项目不存在、照片编号缺失属于永久失败，不进入无限重试。
- `publishJpeg` 继续按 `Pictures/SiteMark/<photoNumber>.jpg` 查找并覆盖，确保崩溃后重跑不会生成重复成片。

为避免删除和后台发布竞态，`captured`、`rendering` 状态不提供编辑或删除；`ready` 和最终 `failed` 才开放相应操作。后台任务开始和结束时必须关闭 Drift 数据库，并在发布前重新确认记录仍存在。

## 界面结构

```text
首页
├─ 全部记录
│  ├─ 项目筛选
│  ├─ 年 / 月 / 日筛选
│  └─ 共用记录卡片列表
├─ 全局设置
│  ├─ 外观与语言
│  ├─ 新项目默认水印
│  └─ 关于与许可
└─ 项目详情
   ├─ 年 / 月 / 日筛选
   ├─ 共用记录卡片列表
   └─ 拍摄表单
      └─ 系统相机 → 入队 → 回到已回填表单
```

项目记录页与全部记录页必须复用同一 `CaptureRecordCard`、`CaptureImagePreview` 和 `CaptureDateFilterBar`，防止状态文案、缩略图和筛选规则分叉。

## 水印排版

Rust 中基础字号从当前值整体乘以 1.20：

- 正文字号：`((width * 0.0312).clamp(31.2, 69.6))`。
- 标题仍为正文的 `1.18` 倍。
- 行高仍为正文的 `1.42` 倍。
- padding、强调色竖条和卡片高度随字号重新计算。
- 卡片宽度保持照片宽度的 62%；若文字宽度超过可用区域，使用现有绘制规则但必须通过 4:3、16:9、竖拍和最低支持尺寸测试，禁止越界或裁切。

## 非目标与平台边界

- 不改为 App 内自研相机，不加入第三方相机界面。
- 不加入云同步、账号、广告、分析、后台定位、网络权限或广泛媒体权限。
- 不实现 iOS、图库导入、自由拖拽水印、批量编辑或跨设备同步。
- Android WorkManager 的启动时刻由系统决定；本版本保证前台不等待并保证任务可恢复，不承诺毫秒级立即执行。
- 用户在系统设置中“强行停止”应用后，后台调度会暂停到下次手动打开；普通返回、划掉最近任务、进程回收和重启设备均纳入恢复测试。

## 验收与交付

- 自动测试覆盖 schema 迁移、回填、入队、幂等、三次重试、筛选边界、设置持久化、预览状态和水印尺寸。
- 真机连续拍摄至少 10 张，下一张拍摄不等待上一张水印完成。
- 拍完后划掉 App、重新打开和重启手机，未完成记录最终进入 `ready`，相册中无重复照片。
