# SiteMark 流畅度优化设计

## 目标

在不改变现有功能、界面文案和离线边界的前提下，减少列表滚动、批量选择、筛选、水印滑块、照片详情和存储统计中的重复工作，并为大量工程记录建立可验证的性能基线。

## 方案选择

考虑过三种方案：

1. **局部缓存与共享数据源（采用）**：保留现有 Flutter、Riverpod、Drift 架构，缓存稳定 Future/Stream，把日期筛选放到内存，缩小重建范围。改动集中、风险最低。
2. **全面改成 Riverpod 页面控制器**：状态边界更统一，但会重写多个成熟页面，回归风险和审查成本较高。
3. **把列表状态物化到新数据库表**：查询最快，但会产生额外同步状态，不适合当前数据规模。

## 设计

### 1. 水印设置

- 页面初始化时只读取一次项目。
- 滑块拖动仅更新本地值，不重新创建项目查询。
- 保存逻辑和现有取值范围保持不变。

### 2. 照片列表与选择

- `CaptureRecordCard` 缓存原图状态 Future，仅在照片 ID、原图路径或删除时间变化时重新检查。
- `CaptureImagePreview` 改成有状态异步解析，禁止在 `build()` 中调用 `existsSync()`。
- 图片输出目录在应用生命周期内只解析和创建一次。
- 页面选择变化不再重新订阅数据库流；列表仅重建当前可见卡片。

### 3. 数据流与筛选

- “全部记录”只订阅一次全部照片和一次项目列表，项目及年月日筛选由内存中的完整列表计算。
- 项目详情只订阅一次该项目全部照片，日期筛选由内存计算。
- Stream 在页面状态初始化时创建，不随 `setState()` 重建。
- 后台处理中每个页面最多保留一个照片轮询器。

### 4. 图片解码

- 96×96 列表缩略图维持小尺寸解码。
- 详情页普通预览按屏幕宽度和设备像素比计算解码宽度，并设置上限。
- 用户点击全屏放大时才读取完整分辨率图片。

### 5. 存储统计

- 存储统计 Provider 保持缓存，只有用户刷新、清理导出文件或数据库相关操作后才失效。
- 文件大小读取采用固定并发上限，避免逐个串行读取，也避免一次启动过多 I/O。

### 6. 数据库索引

- 数据库升级到 v5。
- 增加 `status`、全局拍摄时间、`project_id + 拍摄时间` 索引。
- 新建数据库和 v4 升级数据库都创建相同索引，升级不修改已有记录。

### 7. 性能验证

- 增加回归测试，证明滑块重建不会重复查项目、卡片重建不会重复检查文件、筛选不创建第二个照片流、输出目录只初始化一次。
- 增加 1000 条记录的数据测试，验证筛选结果和索引存在。
- 保留现有真机集成测试；Profile 模式记录首页滚动、全选、筛选、详情打开和拍照启动分段耗时，不把模拟器绝对耗时作为发布门槛。

## 兼容与错误处理

- 不新增网络、账号、广告或第三方相机依赖。
- 文件缺失继续显示现有占位提示，不因异步检查闪退。
- 缓存键包含会影响结果的照片字段，后台状态变化后能够自动刷新。
- 数据库索引使用 `IF NOT EXISTS`，迁移可重复执行。

## 验收标准

- 现有功能和 196 个 Flutter 基线测试不回退。
- `flutter analyze`、Flutter 全量测试、Rust 测试、Android 单元测试和 debug APK 构建通过。
- 记录页每个页面只有一个照片汇总订阅；页面重建不重复解析输出目录或检查同一照片原图。
- 详情页普通预览不再无上限解码完整原图。
- v4 数据库升级后数据保留，v5 性能索引齐全。
