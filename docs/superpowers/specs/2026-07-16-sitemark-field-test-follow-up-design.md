# SiteMark 真机测试跟进设计

## 背景与目标

SiteMark v0.2.0 合并后，真机测试暴露出三组影响现场使用的问题：系统相机入口被定位请求阻塞超过 5 秒；水印字号不可调、背景宽度固定且英文界面仍生成中文标签；项目记录和全部记录的筛选控件占用空间过大，并且缺少筛选后的批量管理能力。

本轮改造继续保留“系统相机、无广告、无云端、本地处理”的产品边界，目标是：

- 点击拍摄时不再等待定位，应用立即调用系统相机。
- 优先复用原图 EXIF 中的 GPS；EXIF 不可用时使用与该次拍摄绑定的异步定位结果。
- 字体大小可由用户调节，水印背景宽度按实际内容自适应，中英文界面生成对应语言的水印。
- 项目页的年、月、日筛选和全部记录页的项目、年、月、日筛选均保持单行紧凑布局。
- 筛选结果可以多选导出、再次保存至相册、清理原图或完整删除。
- 列表直接显示原图状态，详情页可以查看原图、查看文件信息并执行两种删除操作。

## 已确认的产品决策

### 定位来源与处理顺序

采用“EXIF 优先、异步定位兜底”的混合方案：

1. 用户点击拍摄时，应用创建相机目标和待拍摄记录；只有定位权限已经授予时才发起一次性定位请求，并且不等待结果。
2. 应用立即调用系统 `ACTION_IMAGE_CAPTURE`。
3. 系统相机返回成功后，前台只完成原图有效性确认和记录状态持久化，然后立即恢复拍摄按钮。
4. 后台协调器读取原图 EXIF GPS；存在有效 GPS 时优先采用，并忽略较晚返回的应用定位。
5. EXIF 没有 GPS 时，使用点击拍摄时已经发起且与该记录绑定的定位结果。
6. 两者均不可用或定位超时，记录明确标记为“无可用位置”，水印继续生成，不因定位无限等待。

定位解析和水印处理在后台协调，但存在确定的先后依赖：位置来源先解析为“已确定”或“不可用”，再将照片加入持久化水印队列。这个依赖只影响该照片何时生成水印，不阻塞界面，也不阻止用户继续拍摄下一张。

定位权限只能在进入拍摄表单后的独立引导中预先申请，拍摄按钮路径绝不弹出权限对话框，避免权限窗口与系统相机争用 Activity。用户尚未授权、拒绝授权或选择不处理引导时，拍摄按钮仍立即工作，该次拍摄只使用 EXIF。应用不申请持续后台定位权限，不增加网络权限。外部系统相机处于前台时，一次性定位是否继续返回由 Android 和设备决定，因此 EXIF 是首选，应用定位只是兜底。

### 水印设置

- 全局设置的“新项目默认水印”和项目水印设置均增加字体大小。
- 字体大小使用 80%–160% 滑块，步进 5%，100% 对应 v0.2.0 当前视觉尺寸。
- 全局默认值只复制给新建项目，不追溯修改已有项目；项目设置控制该项目后续照片。
- 水印卡片宽度不再固定为图片宽度的 62%，而是根据标题和各字段渲染后的实际宽度计算。
- 卡片宽度包含强调色竖条、内边距和安全边距；最小宽度只由实际标题、字段和内边距决定，不设固定图片占比，最大宽度限制在图片宽度约 92%。
- 内容超过最大宽度时按中英文各自的断行规则换行，不通过无限加宽或静默丢弃字段解决。
- 拍摄时保存当前解析后的语言代码。中文界面使用中文标签，英文界面使用英文标签；后台任务不能因为用户随后切换应用语言而改变该照片的水印语言。
- 已经生成完成的旧照片不自动重绘；升级时没有语言快照的旧记录按中文兼容。

### 筛选与编辑模式

- 项目详情页使用一行三个等宽紧凑筛选按钮：年、月、日。
- 全部记录页使用一行四个紧凑筛选按钮：项目、年、月、日；项目名称过长时省略显示，完整名称在菜单中展示。
- 两个页面的右上角均提供“编辑”；进入后变为“完成”，记录卡片显示多选框。
- 支持选择单张、选择多张和全选当前筛选结果。
- 修改任一筛选条件时清空当前选择，避免对已经不可见的照片误操作。
- `captured` 和 `rendering` 状态仍禁止删除，避免与后台发布产生竞态；`ready` 可以执行全部操作，最终 `failed` 记录仅开放原图清理和完整删除。

### 原图状态与删除语义

每张记录卡片必须显示以下一种原图状态：

- 原图已保留：数据库记录存在，且原图文件可读取。
- 原图已清理：用户主动执行过原图清理。
- 原图异常丢失：用户没有执行清理，但文件已经不存在或不可读取。

批量编辑和照片详情复用同一套业务服务，并提供两种明确的删除行为：

- “清理原图”或详情页“删除原图”：只删除未加水印的私有原始文件，保留水印照片、拍摄记录、照片编号、原图 SHA-256 和导出所需字段，并写入原图清理时间。
- “完整删除”或详情页“删除全部”：删除原图、应用私有水印文件、MediaStore 中已发布的水印照片以及拍摄记录。操作前显示数量和不可恢复提示。

原图清理后，查看原图和所有依赖原图的重新渲染功能变为不可用；水印照片、记录详情和已有校验信息继续可查看。完整删除成功后，详情页返回上一级并刷新列表。

## 技术设计

### 数据库迁移

Drift schema 从 3 升级到 4，并保持现有项目、照片、编号、哈希和 MediaStore URI 不变：

- `projects.watermarkFontScale REAL NOT NULL DEFAULT 1.0`。
- `app_settings.defaultWatermarkFontScale REAL NOT NULL DEFAULT 1.0`。
- `captures.watermarkLocaleCode TEXT NOT NULL DEFAULT 'zh'`。
- `captures.locationResolution TEXT NOT NULL DEFAULT 'resolved'`，新拍摄记录使用 `pending`，最终进入 `resolved` 或 `unavailable`。
- `captures.originalDeletedAt DATETIME NULL`，用于区分主动清理与异常丢失。

数据库新增按记录更新位置解析状态的方法。原图状态不单独存储重复布尔值：主动清理由 `originalDeletedAt` 判断，文件存在性在列表摘要和详情读取时核验。

### 相机与位置协调

当前性能问题的根因是 `CaptureWorkflow.capture()` 在 `launchCamera()` 前同步等待最长 10 秒的 `requestCurrentLocation()`。改造后必须保证调用顺序可由测试验证：

```text
创建目标与 pending 记录
├─ 已授权时启动一次性定位 Future（不 await）
└─ 立即 launchCamera
       ↓ 相机返回
持久化 captured 并恢复前台交互
       ↓
CaptureLocationCoordinator
├─ 读取 EXIF GPS
├─ 无 EXIF 时等待该记录的定位 Future（有限超时）
├─ 持久化位置或 unavailable
└─ enqueue WorkManager
       ↓
CaptureProcessor 渲染并发布水印照片
```

`CaptureLocationCoordinator` 只负责位置来源选择和入队，不执行图片渲染。若应用在相机返回后的协调阶段被系统终止，启动恢复会重新读取 EXIF；EXIF 仍不存在时标记位置不可用并入队，不用后来取得的位置冒充拍摄地点。

性能埋点记录从点击拍摄到调用 `launchCamera` 的应用侧耗时。验收要求该路径不等待定位、EXIF、哈希、Rust 渲染或 MediaStore 发布；应用侧 P90 目标不超过 300ms。系统相机真正显示的时间仍受设备和系统相机影响，但不得再出现由本应用 10 秒定位超时造成的延迟。

### EXIF 与水印接口

图像层新增只读元数据接口，返回原图宽高、格式和可选 GPS。GPS 必须同时存在有效纬度和经度，且范围合法；无效或不完整 EXIF 视为不存在，不使任务失败。

Rust `RenderPhotoRequest` 增加：

- `fontScale`
- `localeCode`

Rust 根据 `localeCode` 选择中文或英文标签集合，包括现场记录、位置、内容、拍摄人、编号、时间、地址、坐标和备注。动态卡片布局先生成最终行文本、测量每行宽度，再确定卡片宽高并绘制；短文本收窄，长文本在最大宽度内换行。所有字号、行高、内边距和强调条尺寸随 `fontScale` 成比例变化。

### 记录摘要和详情元数据

共用的记录摘要模型增加原图可用状态。列表不得同步解码全分辨率图片，只检查必要的文件状态并继续使用现有缩略图策略。

详情页异步读取并展示：

- 水印照片文件大小、分辨率、JPEG 格式和相册保存状态。
- 原图文件大小、分辨率、格式和“已保留/已清理/异常丢失”状态。
- 拍摄时间、项目、现场字段、坐标、原图 SHA-256 和照片编号。

原图存在时，预览区域支持在“水印照片”和“原图”之间切换，并沿用现有全屏缩放预览。原图不可用时保留状态说明，不显示无效切换入口。

### 批量操作服务

新增以 `captureIds` 为输入的统一服务，供项目页、全部记录页和详情页复用：

- 导出：只接受 `ready` 记录。跨项目选择时生成一个 ZIP，根目录包含清单文件，每个项目建立独立文件夹并保留照片编号。
- 再次保存至相册：将每条 `ready` 记录的私有水印文件重新发布到既有 `Pictures/SiteMark` 位置，并更新实际返回的 `publishedUri`；沿用同名覆盖规则，不制造重复照片。
- 清理原图：逐条删除原图并写入 `originalDeletedAt`，重复执行保持幂等。
- 完整删除：逐条删除原图、私有水印文件和已发布照片，全部外部文件清理成功后再删除数据库记录；失败项保留记录并汇总原因，避免界面显示成功但相册仍残留照片。

批量任务显示已完成数量和总数量，结束后汇总成功、跳过和失败项。破坏性操作必须二次确认，不提供无确认的一键清空。

## 界面结构

```text
项目记录 / 全部记录
├─ 单行紧凑筛选栏
├─ 编辑 / 完成
├─ 记录卡片
│  ├─ 缩略图与处理状态
│  └─ 原图已保留 / 已清理 / 异常丢失
└─ 编辑模式底部操作栏
   ├─ 导出
   ├─ 保存至相册
   ├─ 清理原图
   └─ 完整删除

照片详情
├─ 水印照片 / 原图切换预览
├─ 文件与拍摄信息
├─ 删除原图
└─ 删除全部
```

筛选栏在 360dp 宽度下必须保持单行，不允许横向溢出。按钮采用紧凑高度和较小水平内边距，但点击区域不得小于 Material 建议的最小触控尺寸。

## 错误处理和恢复

- 定位被拒绝、关闭或超时：继续拍摄和生成水印，位置显示为不可用。
- EXIF 无 GPS 或损坏：回退到该次拍摄的一次性定位，不把元数据错误当成照片处理失败。
- 原图在处理前被异常移除：保持永久失败记录并显示“原图异常丢失”。
- 原图清理与完整删除只能在后台渲染结束后执行。
- 再次保存或导出遇到缺少水印文件时跳过该项并报告，不使用原图临时伪造水印成片。
- 批量操作逐条持久化结果；应用中断后，已经完成的删除不重复报错，未完成项可以再次执行。

## 测试与验收

### 自动测试

- schema 3→4 迁移保留现有数据并正确填入默认值。
- 未完成的定位 Future 不能阻止 `launchCamera` 被调用。
- EXIF GPS 优先于应用定位；无 EXIF 时正确回退；两者不可用时仍然入队。
- 相机返回后前台立即恢复，定位协调和 WorkManager 入队不阻止下一次拍摄。
- 中文、英文标签集完整，英文渲染结果不包含中文固定标签。
- 80%、100%、160% 字号以及短文本、长中英文文本均不越界、不裁切。
- 360dp 下三筛选和四筛选保持单行，级联逻辑不退化。
- 选择在筛选变化后清空；跨项目 ZIP 按项目分目录。
- 清理原图保留水印照片、记录和哈希；完整删除清理所有文件和记录；两种操作均幂等。
- 详情页与批量编辑调用相同删除服务并呈现一致结果。

### 真机验收

- 在定位冷启动、定位已缓存、定位关闭和权限拒绝四种情况下分别连续启动系统相机 10 次，不出现应用等待 5–10 秒后才调用系统相机。
- 系统相机开启和关闭照片位置标签时，分别验证 EXIF 优先和应用定位兜底。
- 连续拍摄至少 10 张，前一张定位和渲染不影响下一张进入系统相机。
- 切换为 English 后新照片使用英文水印，切回中文后新照片使用中文水印。
- 调整字体大小并使用短字段、长中文和长英文拍照，背景宽度随内容变化且不超出照片。
- 在项目页和全部记录页验证单行筛选、批量选择、跨项目 ZIP、再次保存、清理原图和完整删除。
- 详情页验证原图/水印切换、文件大小和分辨率，以及两个独立删除按钮。

## 非目标

- 不改为应用内自研相机，不接入第三方相机 SDK。
- 不申请持续后台定位，不新增云端定位、地图或地址搜索。
- 不自动重绘已经发布的旧照片。
- 不提供删除后的回收站或云端恢复。
- 不在本轮加入自由拖拽水印、图库导入或跨设备同步。
