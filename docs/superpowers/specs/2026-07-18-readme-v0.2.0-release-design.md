# SiteMark v0.2.0 Release README Design

**Date:** 2026-07-18
**Status:** Approved for planning
**Scope:** Update the repository README for the first signed prerelease and record the next P0/P1 product improvements.

## Goal

Turn `README.md` into a Chinese-first download and onboarding page for ordinary
Android users while retaining contributor and architecture information below the
user-facing content. Every statement must match the merged `main` branch and the
published `v0.2.0` prerelease.

## Verified release facts

- Repository: `WikG1018/site-mark`.
- Release: `v0.2.0`, published as a GitHub prerelease from tag `v0.2.0`.
- Minimum supported version: Android 12 / API 31.
- Signed artifacts:
  - `sitemark-v0.2.0-arm64.apk` for most current Android phones;
  - `sitemark-v0.2.0-universal.apk` as the compatibility fallback;
  - `SHA256SUMS.txt` for integrity verification.
- Current automated evidence after PR #5: 185 Flutter tests and 20 Rust tests,
  plus clean Flutter analysis, Rust format/lint, Android unit tests, and signed
  release APK construction in GitHub Actions.
- The watermark no longer displays the photo number. The number remains stored
  for filenames, records, traceability, regeneration, and export.

## Audience and information architecture

The primary audience is a phone user who wants to decide whether the app is
suitable, download it, install it, and avoid losing data. Contributors remain a
secondary audience.

The README order will be:

1. Product name, one-line promise, badges, and current prerelease status.
2. Direct download card/table with arm64, universal, checksums, Android version,
   and a concise recommendation.
3. Installation and upgrade instructions.
4. Product advantages and current screenshots.
5. Quick-start workflow.
6. Data storage, permissions, privacy, and uninstall warning.
7. Record management, original-photo lifecycle, and export behavior.
8. Known limitations and the approved P0/P1 improvement roadmap.
9. Technical architecture, local build instructions, contribution links, and
   license.

## Download and installation content

The README will link directly to the three `v0.2.0` release assets and also link
to the release page. It will explain that most modern phones should install the
smaller arm64 package and use the universal package only when architecture is
unknown or the arm64 package cannot be installed.

Installation guidance will cover Android's "allow unknown apps" prompt. Upgrade
guidance will distinguish:

- installing a newer APK signed with the same release key, which preserves app
  data; and
- replacing an older debug-signed build, which Android may reject because the
  signing certificates differ.

The README must not recommend uninstalling as the first solution to a signature
conflict. It must first warn the user to export important projects and originals.

## Data-safety contract

The data warning must be visible near installation rather than buried in the
technical section:

- project records, the local database, and retained originals live in the app's
  private storage;
- uninstalling SiteMark clears that private storage;
- watermarked JPEGs already published to `Pictures/SiteMark` are outside the
  private database lifecycle and normally remain;
- users should export required projects and originals before uninstalling,
  changing signing channels, resetting the phone, or migrating devices.

The wording must avoid implying that SiteMark currently provides full restore or
device migration, because that capability is a roadmap item.

## Feature corrections and additions

The README will describe the merged behavior, including:

- manufacturer/system camera launch through Android Intent;
- immediate return to the capture form with retained site/location/work/author
  fields and cleared notes;
- persistent background rendering with automatic list refresh;
- project search and year/month/day/project filters;
- previews, details, file size, and original-retention state;
- batch selection for export, resaving, clearing originals, or complete deletion;
- per-photo original viewing and deletion plus complete record deletion;
- global settings, Chinese/English, themes, and new-project watermark defaults;
- project watermark font scale, card opacity, position, and accent color;
- project-prefixed photo filenames while omitting the number from the visible
  watermark;
- ZIP export with watermarked images, CSV, JSON manifest, and optional originals.

Any legacy statement that the visible watermark contains a photo-number row will
be removed.

## Permissions and privacy

The permission table will match the packaged release behavior and include the
notification permission contributed for background work. It will continue to
state that location is optional and that the release does not request `CAMERA`,
`INTERNET`, broad media access, background location, or legacy storage access.

The README will distinguish "no network permission" from "the About page can
open a GitHub URL in the browser": the browser, not SiteMark, performs that
network access.

## Screenshot strategy

The existing screenshot set predates the merged PR #4/#5 behavior. In particular,
the current watermark image visibly contains the removed number row, and the home
screen does not show current actions.

All screenshots used by the README must come from the current `main` build with
synthetic project data. At minimum the gallery must accurately show:

- the current project home/search/settings entry points;
- the retained capture form;
- the external Android system camera;
- the current watermarked result without a visible photo number.

If a current screenshot cannot be reproduced and visually verified, remove the
stale image instead of retaining misleading evidence. Captions must disclose the
emulator/synthetic-data origin and that OEM camera interfaces differ.

## Approved improvement roadmap

Only P0 and P1 items are included in this README update.

### P0

- Build and publish a physical-device compatibility matrix covering Xiaomi,
  OPPO, vivo, Honor, Samsung, and Pixel, with Android version, camera-return
  timing, background-processing behavior, and known workarounds.
- Add complete project backup and restore so users can safely uninstall, change
  phones, or recover private originals and the project database.

### P1

- Add a local diagnostic-package export containing device/OS/app versions,
  camera-launch/return timings, and background task failures, excluding photo
  content by default.
- Show private-storage usage and support project/date-based original cleanup.
- Replace automatically generated release bodies with user-oriented release
  notes covering installation choice, changes, upgrade warnings, known issues,
  and checksums.

P2 accessibility, large-text, landscape, and extreme-text initiatives are
explicitly excluded at the user's request.

## Error handling and honesty rules

- A broken release asset link is a release-blocking README defect.
- A stale test count, permission claim, screenshot, or version claim must be
  corrected or removed.
- Known device behavior must not be described as verified unless there is a
  recorded physical-device result.
- SiteMark's SHA-256 metadata must be described as integrity/traceability data,
  not judicial authentication or third-party notarization.

## Verification and acceptance

Before publishing:

1. Verify all direct release links return the expected assets.
2. Confirm README version, Android requirement, artifact names, hashes, and test
   counts against GitHub Release, workflows, and the current codebase.
3. Confirm all local Markdown links and image paths exist.
4. Search for obsolete claims such as "no signed APK", "wait for the first
   prerelease", visible watermark numbering, and old test counts.
5. Render or inspect the README as GitHub Markdown and visually inspect every
   screenshot and table.
6. Run documentation-relevant repository checks and ensure the branch contains
   only the approved README/design/screenshot changes.
7. Commit on `agent/readme-v0.2.0-release`, push it, and open a draft PR against
   `main` for final review rather than writing directly to `main`.
