# SiteMark 处理状态刷新与照片命名修复设计

**日期：** 2026-07-17  
**状态：** 已确认  
**适用分支：** `feat/sitemark-field-test-follow-up`

## 背景

真机测试发现，WorkManager 在后台完成水印处理后，项目记录列表仍停留在“等待处理”或“处理中”，必须离开页面后重新进入才会显示最终状态。同时，现场照片的数据库编号和相册文件名缺少项目名称，不利于跨项目整理；水印卡片中的编号又占用显示空间，没有现场阅读价值。

本轮保持现有水印设置语义不变：主页总设置仍然只决定新建项目的默认水印参数，已有项目继续使用各自的项目水印设置。本轮不修改字体大小或透明度的保存、继承规则。

## 目标

- 后台任务把照片更新为“已完成”或“失败”后，当前打开的列表和详情自动刷新。
- 新拍照片的编号和相册文件名包含项目名称。
- 水印卡片不再显示照片编号。
- 已有照片、已有编号和相册中的已有文件不自动改名。

## 方案比较

### 方案一：编号源头统一生成项目化编号（采用）

在照片从系统相机返回并执行 `markCaptured` 时读取项目名称，直接生成包含项目名称的持久化 `photoNumber`。后台发布、详情、导出和相册文件名继续复用同一个编号。这样各处不会出现数据库编号与文件名不一致。

### 方案二：只修改相册文件名（不采用）

保留数据库旧编号，只在发布相册时拼接项目名称。实现较小，但详情、导出清单和相册文件名会使用两套标识，后续排查困难。

### 方案三：新增独立文件名字段（不采用）

可以分别管理业务编号和文件名，但需要 schema 迁移、绑定更新和额外兼容逻辑。本轮两者规则一致，没有引入新字段的必要。

## 新照片编号与文件名

### 格式

新拍照片使用：

```text
{安全项目名称}-SM-{yyyyMMdd}-{三位流水号}
```

例如：

```text
东区厂房改造-SM-20260717-001
东区厂房改造-SM-20260717-001.jpg
```

数据库 `photoNumber` 保存不带扩展名的第一种形式；发布到相册时沿用该编号并生成 `.jpg` 文件名。项目内的日期流水规则保持不变，每个项目每天从 `001` 开始递增。

### 项目名称安全化

编号中的项目名称按以下固定规则处理：

1. 去除首尾空白。
2. 将控制字符和 `/ \ : * ? " < > |` 替换为下划线。
3. 连续空白或连续下划线折叠为单个下划线。
4. 项目名称部分最多保留 60 个 Unicode 码点。
5. 截断后去除处理结果首尾的句点、空格和下划线。
6. 如果处理后为空，使用 `Project` 作为兜底名称。

安全化只作用于照片编号和文件名，不修改项目在界面及数据库中的原始名称。

### 生效范围

- 只对修复版本安装后新拍并完成相机返回的照片生效。
- 已有照片的 `photoNumber` 不迁移。
- 重新生成已有照片时继续使用其已有编号，不借机改名。
- 不扫描、不重命名也不重复发布相册中的历史文件。

## 水印卡片

Rust 渲染器从逻辑水印行中删除“编号”字段及其值。中文和英文水印都不显示编号；其余项目名称、位置、内容、拍摄人、时间、地址、坐标和备注保持现状。

自适应布局继续根据最终可见行重新测量宽高。删除编号行后，卡片高度应自然缩小，不保留空行。照片编号仍保留在数据库、详情页、批量导出和相册文件名中。

## 后台处理状态自动刷新

### 根因

前台和 WorkManager 后台任务分别创建 `AppDatabase`。默认 Drift 连接不会可靠地把后台连接的表更新通知给前台查询流，所以后台已经写入 `rendering`、`ready` 或 `failed` 时，当前页面的 `StreamBuilder` 仍可能保留旧快照。

### 主路径：跨 isolate 数据库通知

生产数据库通过 `DriftNativeOptions(shareAcrossIsolates: true)` 打开，使同一 Flutter 运行环境中的 Drift 连接共享更新通知，并由现有 watch 查询立即驱动界面刷新。

### 兜底路径：仅处理期间轮询

考虑到 Android 厂商实现和后台 Flutter 引擎可能无法共享通知，记录监听增加条件轮询：

- 当监听结果包含 `captured` 或 `rendering` 状态时，每秒从 SQLite 重新读取一次同一查询。
- 新结果与当前结果不同时向界面发出更新。
- 所有记录进入 `ready`、`failed` 或其他非处理状态后立即停止定时器。
- 页面取消订阅时立即停止定时器。
- 没有等待处理照片时不启动轮询，不产生常驻数据库读取。
- 单次轮询遇到临时数据库忙或读取失败时保留当前画面，下一周期继续尝试；不会终止原有 Drift 监听流。

该刷新策略覆盖项目记录列表、全部记录列表和照片详情。筛选条件保持不变，刷新不能重置当前筛选或编辑选择状态。

## 数据流

```text
系统相机返回
  -> markCaptured 读取项目名称
  -> 生成 项目名称-SM-日期-流水号
  -> captured（前台列表开始条件轮询）
  -> WorkManager: rendering
  -> Rust 渲染不含编号的水印
  -> MediaStore 使用项目化编号发布 JPEG
  -> ready / failed
  -> Drift 通知或下一次轮询更新当前页面
  -> 条件轮询停止
```

## 错误处理

- 项目在编号生成前已被删除时，沿用现有外键和缺失项目错误处理，不生成无归属编号。
- 文件名安全化必须是纯函数，所有平台使用同一结果。
- 后台处理失败仍写入 `failed` 和失败原因；自动刷新后界面显示失败及重试入口。
- 状态刷新只读取数据库，不重新入队、不重复渲染、不改变处理次数。

## 测试

### Dart / Drift

- 新拍照片生成 `项目名称-SM-yyyyMMdd-001`，同项目同日继续递增。
- 项目名称非法字符、连续空白、空结果和超长名称得到稳定安全结果。
- 历史照片和重新生成流程保留原编号。
- CaptureProcessor 把项目化编号传给 MediaStore 发布接口。
- 使用同一 SQLite 文件的第二个独立数据库连接更新状态；前台监听即使没有本地 Drift 通知，也会在条件轮询周期内从 `captured` 更新为 `ready`。
- 进入终态后不再继续轮询，取消订阅后不再读取。
- 项目页、全部记录页和详情页在状态变化后保留筛选及选择状态。

### Rust

- 中文和英文逻辑水印行都不包含编号标签或照片编号。
- 删除编号后渲染结果无空行，卡片尺寸仍在图片边界内。
- 字体大小、透明度、语言和自适应宽度现有测试继续通过。

## 验收标准

- 新照片详情中的编号包含安全化项目名称。
- 相册 JPEG 文件名与详情编号一致并包含项目名称。
- 新照片水印不显示“编号”或 `Number` 行。
- 已有照片没有被自动改名或重复保存。
- 保持列表页面打开时，后台任务完成后无需返回或手动刷新即可显示最终状态和预览图。
- 没有处理中照片时不运行周期性刷新。
- 首页总设置仍只作为新项目默认值，本轮不改变该产品规则。
