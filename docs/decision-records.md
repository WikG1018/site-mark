# SiteMark 关键技术决策记录

> 状态：当前决策索引
> 适用版本：v0.2.0
> “已接受”表示当前代码应遵守；如果以后改变，必须新增替代决策并标记原决策状态。

## D-001 使用系统相机，而不是应用内相机

**状态：已接受**

通过 Android `ACTION_IMAGE_CAPTURE` 和一次性 ContentProvider URI 调用手机系统或
厂商相机。SiteMark 不申请 `CAMERA`，也不集成第三方相机 SDK。

理由是保留用户熟悉的厂商拍照体验和设备原生能力。代价是不同厂商相机的启动耗时、
返回行为和画面功能存在差异，因此必须通过真机矩阵验证，不能只依据模拟器结论。

## D-002 保持完全离线

**状态：已接受**

发布 APK 移除 `INTERNET` 和 `ACCESS_NETWORK_STATE`，不加入账号、广告、分析、
云同步或远程 API。GitHub 链接只交给外部浏览器打开。

这样可以降低工程资料外传风险并保持功能可解释；代价是没有自动云备份、跨设备同步和
应用内版本检查。

## D-003 私有保留原图，公开发布水印成片

**状态：已接受**

原图和渲染中间文件保存在应用私有目录，水印成片通过 MediaStore 发布到
`Pictures/SiteMark`。应用不申请广泛媒体权限。

这兼顾重新生成和系统相册可用性；代价是卸载会删除私有原图和数据库，而相册成片通常
继续存在。README 必须持续提示这一生命周期差异。

## D-004 全分辨率处理使用持久化串行后台队列

**状态：已接受**

相机返回后前台只完成必要持久化，EXIF、哈希、水印和相册发布通过 WorkManager 处理。
所有任务进入同一串行链，处理器以 `captureId` 为输入且可幂等重跑，临时错误最多重试
3 次。

该决策提高连拍效率并控制内存峰值；代价是完成时间受 Android 调度影响，强行停止应用
后必须再次启动才能恢复。

## D-005 定位优先读取原图 EXIF

**状态：已接受**

位置来源顺序为“原图 EXIF GPS → 已授权的单次前台定位 → 不可用”。定位授权在拍摄
表单中说明并请求，授权后不重复提示；拒绝或超时不阻止拍照。禁止后台定位。

这样优先使用照片自身携带的位置，并让定位与后台处理并行；代价是部分系统相机不写
EXIF GPS，且用户拒绝定位时水印只能明确显示位置不可用。

## D-006 Rust 负责图像和归档核心

**状态：已接受**

EXIF 方向、SHA-256、全分辨率水印和 ZIP/CSV/JSON 由 Rust 实现。Flutter 与 Rust
通过 flutter_rust_bridge 传递路径和结构化请求，不跨 FFI 复制整张图片。

这样把计算密集且可测试的逻辑集中在同一核心；代价是发布构建需要 Rust/NDK 工具链，
Flutter、Rust 和 Android 三层文件名及参数校验必须保持一致。

## D-007 SQLite 是记录状态的唯一事实来源

**状态：已接受**

项目、拍摄状态、路径、编号、位置、哈希和设置写入 Drift/SQLite。前台不依赖内存中的
临时任务状态判断是否完成；后台 isolate 更新数据库后，界面通过监听和处理中条件轮询
获得变化。

该决策支持进程恢复和统一筛选；代价是数据库 schema 变更必须提供保留用户数据的迁移。

## D-008 照片编号保留，但不绘制在水印上

**状态：已接受**

编号继续用于数据库、文件名、相册发布、CSV 和 JSON manifest，以保证记录唯一性和
可追溯性；水印画面不显示“编号/Number”行，减少现场成片的视觉负担。

修改编号格式需要同时更新 Dart 生成规则、Android 接收校验、Rust 归档校验和跨层测试。

## D-009 导出是外部归档，不是应用备份恢复

**状态：已接受**

v0.2.0 ZIP 包含水印照片、CSV、JSON manifest 和可选原图，用于外部留档和交付。
当前没有导入 ZIP、重建数据库或恢复项目的入口。

因此界面和文档不得把“导出”写成“可恢复备份”。完整项目备份与恢复是 P0 后续功能，
在实现导入、版本迁移和冲突处理前不能宣称支持换机恢复。

## D-010 水印采用受约束的自适应模板

**状态：已接受**

用户可以调整左右位置、透明度、字体比例和有限强调色；卡片宽度根据实际文字测量，
水印语言取拍摄时界面语言。不提供自由拖拽和脚本化模板。

该决策在可配置性和稳定输出之间取平衡，便于对横拍、竖拍、长文本和中英文建立确定性
测试。

## D-011 文件名采用跨层一致的安全契约

**状态：已被 D-013 替代**

项目名称使用黑名单替换空白、控制字符、`~` 和路径/文件系统禁止字符，保留括号、点号、
连字符、`&` 等安全标点；项目 ID 使用 ASCII 白名单。`~` 作为专用字段分隔符，最终
JPEG 名称不超过 255 个 UTF-8 字节。

这样既保留可读项目名称，又避免目录穿越、跨字段碰撞和多字节截断。Dart、Android 和
Rust 的规则不一致视为阻止发布的问题。

## D-012 最低支持 Android 12

**状态：已接受**

当前 `minSdk` 为 31，发布目标覆盖 Android 12 及以上。这样可以围绕现代 MediaStore、
前台服务和权限行为维护较小的兼容面；代价是 Android 11 及更早设备不能安装。

## D-013 新照片使用短编号并禁止项目名称碰撞

**状态：已接受；替代 D-011 的编号组成部分**

新照片编号采用`安全化项目名称-SM-yyyyMMdd-全应用当日序号`，不再包含项目 UUID；
同一天的序号跨所有项目递增。已有记录和已有文件保持原编号，不做迁移或批量重命名。

为了在缩短文件名后继续避免碰撞，新建项目同时比较规范化显示名称和安全文件名键：
忽略首尾空白、连续空白和英文大小写，并拒绝 `A/B` 与 `A:B` 这类生成相同文件名的名称。
安全字符黑名单和 255 字节上限继续沿用。这样提高现场列表和文件管理的可读性，同时把
唯一性约束前移到项目创建边界。

## D-014 存储统计只覆盖应用私有数据

**状态：已接受**

总设置统计私有原图、私有水印成片、本地导出文件、数据库和其他应用文档；系统相册
`Pictures/SiteMark` 明确排除。清理本地导出只删除应用私有 `exports` 目录中的 ZIP。

该边界避免为了统计或清理而申请广泛媒体权限，也避免误删用户已经分享的副本和相册
照片。代价是界面显示的数字不是 SiteMark 相关文件在整台设备上的总占用。

## 决策变更规则

- 不直接删除已经发布版本遵循的决策；
- 新决策必须说明替代哪一条、迁移影响和用户数据处理；
- 权限、存储、导出恢复、相机调用和照片编号契约的变化必须同时更新 README；
- 阶段性 PR 实施步骤保留在 `docs/superpowers/`，不作为当前决策依据。
