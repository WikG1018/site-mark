# 拍摄、后台处理与照片存储生命周期

> 状态：当前设计
> 适用版本：v0.2.0

## 1. 拍摄状态机

```text
pendingCamera ──相机成功──> captured ──后台开始──> rendering ──成功──> ready
      │                         │                     │
      ├─取消：删除临时记录      └─临时错误：最多重试 3 次
      └─错误──────────────────────────────────────────────> failed
```

- `pendingCamera`：已创建相机写入目标，正在等待外部相机返回；
- `captured`：原图有效、编号已分配，正在等待后台处理；
- `rendering`：正在校验原图、生成水印或发布到相册；
- `ready`：水印成片和数据库状态均已完成；
- `failed`：永久错误或第三次临时错误，用户可在保留原图时重新处理。

列表不显示 `pendingCamera`。`captured` 和 `rendering` 通过 Drift 监听加条件轮询刷新，
因此后台 isolate 更新数据库后，前台无需手动刷新即可看到状态变化。

## 2. 前台拍摄顺序

1. 用户点击拍摄时，如果定位已经授权，应用同时发起一次定位读取，但不等待结果；
2. Kotlin 创建仅供本次拍摄写入的 ContentProvider URI；
3. Flutter 先写入 `pendingCamera` 记录，再通过 `ACTION_IMAGE_CAPTURE` 调用系统相机；
4. 用户取消时，临时记录和空目标被清理；
5. 相机成功返回时，数据库写入拍摄时间和照片编号，状态变为 `captured`；
6. 前台立即恢复可操作状态，定位解析和后台入队不阻塞下一次拍摄。

Android 如果在外部相机期间回收 Flutter 进程，启动恢复逻辑会读取持久化的相机目标：
有有效内容则继续处理，没有内容则按取消清理。

## 3. 定位来源与授权

定位按固定优先级解析：

1. 原图 EXIF 中存在有效 GPS 时直接使用；
2. EXIF 没有 GPS 时，使用点击拍摄时已经启动的单次前台定位结果；
3. 两者均不可用时记录为“位置不可用”，仍继续生成水印。

定位权限提示由拍摄表单在合适时机显示。已经授权后不再重复显示提示；未授权时，
拍摄按钮不会暗中触发系统授权框。应用不请求后台定位，也不会因为定位失败阻止拍照。

## 4. 后台处理队列

每张照片以 `captureId` 作为任务输入，追加到同一个 WorkManager 串行工作链，避免两张
全分辨率图片同时渲染造成内存和 IO 峰值。单次处理顺序固定为：

1. 重新读取记录和所属项目；
2. 对已完成或已删除记录幂等退出；
3. 增加处理次数并校验原图路径、拍摄时间和照片编号；
4. 首次计算原图 SHA-256，已有哈希时重新校验；
5. 将状态改为 `rendering`，调用 Rust 生成私有水印 JPEG；
6. 通过 MediaStore 以照片编号为文件名发布或覆盖系统相册成片；
7. 保存返回的 URI，将状态改为 `ready`。

IO 或系统服务临时失败时由 WorkManager 重试，最多 3 次；原图缺失、必要字段缺失或
哈希不一致属于永久失败。应用启动时先恢复未完成的定位，再重新调度
`captured`/`rendering` 记录。普通进程回收和设备重启可恢复；用户在系统设置中
“强行停止”后，需要再次手动打开应用才能恢复调度。

## 5. 三类照片数据

| 数据 | 保存位置 | 用途 | 卸载影响 |
| --- | --- | --- | --- |
| 私有原图 | 应用私有目录 | 重新生成、原图查看、可选导出、哈希校验 | 会删除 |
| 私有水印图 | 应用私有目录的 rendered 区域 | 应用预览、再次保存、导出 | 会删除 |
| 已发布水印图 | `Pictures/SiteMark` / MediaStore | 系统相册和外部应用查看 | 通常保留 |

SQLite 保存路径、MediaStore URI、原图 SHA-256、原图清理时间和业务字段，不把图片
二进制写入数据库。

## 6. 原图和记录操作

| 操作 | 私有原图 | 私有水印图 | 相册成片 | 数据库记录 |
| --- | --- | --- | --- | --- |
| 清理原图 | 删除 | 保留 | 保留 | 保留，并保留编号和 SHA-256 |
| 再次保存至相册 | 不变 | 读取 | 新建或更新 | 更新发布 URI |
| 删除整条记录 | 删除 | 删除 | 删除 | 删除 |
| 修改信息并重新生成 | 必须存在 | 覆盖 | 覆盖 | 保留不可变证据字段 |
| 导出 | 可选包含 | 包含 | 不依赖相册副本 | 只读取，不删除 |

清理原图不可逆，清理后不能重新生成水印或查看原图。详情页和批量编辑共用相同的
生命周期规则，不能出现“列表删了原图但详情仍显示可重新生成”的分叉行为。

## 7. 导出格式和限制

项目导出或所选记录导出只接受 `ready` 记录。ZIP 按项目组织水印照片，并包含
UTF-8 BOM CSV 和版本化 JSON manifest；用户可以选择包含仍然存在的私有原图。
编号、拍摄时间、现场字段、位置和原图 SHA-256 写入结构化清单。

当前导出是供外部留档和交付的归档格式，不是 SiteMark 数据库备份：应用没有 ZIP
导入、项目恢复或换机恢复能力。完整备份与恢复属于后续 P0，而不是 v0.2.0 已实现功能。
